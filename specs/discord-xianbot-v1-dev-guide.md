# Discord 修仙 Bot 一期开发指导文档（给 AI）

> 读者：AI 编码助手 / IDE Agent / Vibe Coding Agent
> 
> 目标：在 **不过度设计** 的前提下，快速实现一个 **Discord 面板式修仙刷子 Bot** 的一期可玩版本。

---

## 1. 项目目标

这是一个运行在 Discord 上的文字修仙 Bot，一期只做最核心、最好玩的闭环：

- 玩家创建角色，随机获得一个 **命格**
- 平时通过 **挂机修炼** 获取修为
- 消耗 **气机** 去 **通天塔冲新高**
- 满足条件后进行 **突破**，提升境界
- 通过 **论道榜** 主动挑战更高排名玩家
- 通过 **公开面板、称号、榜单、播报** 满足虚荣心

一句话定义：

> 一个以境界成长为主线、以通天塔推进为节奏、以论道榜和公开面板承载虚荣心的 Discord 修仙刷子 Bot。

---

## 2. 核心设计原则

### 必须遵守

1. **先做最小可玩版本，不做大而全系统**
2. **境界是主线成长轴**，数值上必须占主导
3. **命格有戏剧性，但不能决定胜负**
4. **通天塔只负责冲新高**，不做扫荡，不做重置重爬
5. **挂机修炼只负责稳定给修为**，不掉核心成长物
6. **论道榜是主动打榜，不做复杂匹配系统**
7. **Discord 面板是公开展示卡，不做网页前端，不做生图系统**
8. **AI 只允许用于播报文案润色，不得参与核心数值逻辑**

### 明确禁止

不要实现以下内容：

- 插件系统
- 事件总线 / 领域事件框架
- 通用 Buff/Effect 引擎
- 技能树
- 职业系统
- 多装备位
- 套装系统
- 炼丹 / 炼器大系统
- 宗门 / 公会
- 交易市场
- 搜打撤玩法
- 扫荡
- 生图面板 / Canvas / Pillow 图片卡
- Redis / MQ / Celery / 微服务拆分
- 可配置 DSL 数值引擎
- “为了以后扩展”而提前抽象的大框架

如果你发现自己正在设计“未来也许会用到”的系统，请立即停止并回到当前一期需求。

---

## 3. 一期玩法范围

### 只做这 4 个核心系统

1. **角色面板**
2. **通天塔**
3. **挂机修炼**
4. **论道榜**

### 附属系统（轻量）

5. **境界突破**
6. **命格**
7. **本命法宝（极简版）**
8. **播报模板**

---

## 4. 核心循环

玩家日常循环：

1. 打开 `/修仙` 查看公开面板
2. 系统按已通关最高层补算挂机修为
3. 玩家有气机时点击登塔，尝试冲新高
4. 新高层首通给成长奖励
5. 卡关时继续挂机积修为
6. 修为满 + 满足塔层条件后突破
7. 用新面板去打论道榜
8. 通过称号、排名、命格和播报获得虚荣心反馈

---

## 5. Discord 交互要求

### 入口命令只保留少数几个

建议实现：

- `/修仙`：打开主面板
- `/登塔`：快捷冲塔
- `/突破`：快捷尝试突破
- `/榜单`：查看榜单
- `/面板 @用户`：查看他人公开面板（可选）

### 交互形式

- 主体交互使用 **Embed + Buttons**
- 玩家点击按钮后，**优先编辑原消息**，不要每次都发新消息刷屏
- 面板默认公开，让别人能看见
- 不要使用 Markdown 表格

### 文案风格

- 简洁、直接、有仙侠味
- 多用 emoji、符号、短句
- 面板是“展示卡”，不是“后台详情页”

---

## 6. 角色面板设计

角色面板优先展示“牌面”，再展示“数值”。

### 第一层：虚荣信息

优先显示：

- 玩家名
- 当前境界
- 命格
- 当前称号
- 论道榜当前名次
- 历史最高名次
- 最高塔层

### 第二层：成长信息

再显示：

- 修为进度
- 气机
- 已通关最高层 / 挂机层
- 本命法宝
- 战力值

### 第三层：最近高光事件

用一句短描述收尾，例如：

- 昨日突破炼气后期，气势正盛
- 方才斩落论道榜第 31 位
- 轮回之后觉醒荒古圣体

### 面板视觉要求

- 使用 emoji + Unicode 进度条
- 不依赖代码块作为主面板
- 排行榜可以使用代码块或紧凑列表
- 进度条建议示例：`▰▰▰▰▱ 84%`
- 气机建议示例：`●●●○○ 3/5`

---

## 7. 境界系统

### 设计目标

境界是主线成长轴，战力占比应当最高。

### 数值权重建议

- 境界：**80%**
- 本命法宝：**15%**
- 其他（命格 / 称号 / 杂项）：**5%**

原则：

> 境界高的人，在大多数情况下应该能压制境界低的人。
> 命格和法宝主要决定同境争锋时的胜负体验。

### 突破条件

突破需要同时满足：

1. 修为达到要求
2. 通关指定塔层守关层

示例：

- 炼气中期：塔 10
- 炼气后期：塔 20
- 炼气圆满：塔 30
- 筑基：塔 40

---

## 8. 通天塔系统

### 定位

通天塔只负责一件事：**冲新高**。

### 规则

- 玩家发起一次登塔，消耗 1 点气机
- 自动连续挑战
- 单次最多推进 3～5 层
- 碰到失败或守关层后立刻停止并结算
- **塔不重爬**
- 永远记录玩家最高已通关层

### 守关层

- 每 5 层或 10 层一个守关层
- 守关层用于制造自然进度墙
- 守关层也是突破资格门槛

### 奖励规则

#### 首次通关新高层 / 首通守关层

给：

- 本命法宝成长材料
- 首通奖励
- 关键成长反馈
- 更高挂机层收益

#### 已通关重复层

- 不再单独手动重复挑战刷材料
- 不做扫荡
- 重复收益由挂机修炼系统承担

---

## 9. 挂机修炼系统

### 核心定义

挂机不是单独地图。

> 挂机 = 自动按玩家“已通关最高层”周期性结算修为。

### 规则

- 每 5 分钟或 10 分钟逻辑上结算一次
- 收益基于当前最高已通关层
- 主收益是 **修为**
- 不掉新的稀有成长物

### 实现建议

优先使用 **懒结算**：

- 记录 `last_idle_at`
- 玩家打开面板 / 登塔 / 突破 / 打榜时补算挂机收益
- 不要做高频全表扫描定时任务

---

## 10. 气机系统

### 作用

限制高价值推进次数，避免玩家 1 分钟爬上百层。

### 规则建议

- 作为登塔行动资源
- 上限建议 5～8
- 自动恢复
- 显示在玩家角色面板上
- 不需要在登塔结算里重点展示

### 实现建议

优先使用 **懒恢复**：

- 记录 `current_qi`
- 记录 `last_qi_recovered_at`
- 玩家交互时补算恢复值

---

## 11. 本命法宝系统（极简版）

### 定位

承载通天塔首通奖励带来的成长快感，但不能压过境界。

### 一期范围

- 只有一件本命法宝
- 有品质
- 有强化等级
- 最多 1～3 个简化属性倾向

### 原则

- 法宝是“同境优势”，不是“主线本体”
- 不要实现多装备、多词条复杂构筑、多段养成链

---

## 12. 命格系统

### 统一命名

一期只做一个系统：**命格**。

不要同时实现“气运 / 体质 / 天赋”三套概念。

### 获取方式

- 玩家创建角色时随机获得 1 条命格
- 一期不做手动挑选
- 一期不做新手无限刷初始

### 命格类型

一期命格只做 3 类：

- **修行型**：挂机修为 +X%
- **战斗型**：直接提升杀伐 / 护体 / 身法中的一项或多项
- **机缘型**：提升通天塔首通时的额外掉落率

### 战斗型命格原则

- 不要实现“登塔战力 +X%”这种黑盒写法
- 战斗型命格应直接影响三维属性
- 命格影响 PvE 与 PvP，不做模式区别对待
- 战斗型允许在三维内部做组合：
  - 单项型
  - 双项型
  - 全项型
- 这属于战斗型内部组合，不算跨类别双效果命格

### 稀有度建议

- 普通
- 稀有
- 史诗
- 传说

### 数值档位建议

采用：

- 普通：+3%
- 稀有：+6%
- 史诗：+9%
- 传说：+12%

说明：

- 修行型与机缘型直接使用完整档位
- 战斗型中的单项 / 双项 / 全项应按总档位分配，避免全项型成为无脑最优解

### 轮回重修

允许玩家为更换命格付出巨大代价。

建议规则：

- 每天可进行 1 次轮回重修（按自然日刷新）
- 轮回后重新随机命格
- **重置战力相关主线进度**
- 但不删除账号身份

建议重置：

- 当前境界
- 当前修为
- 当前塔层
- 当前法宝成长
- 当前论道即时排名状态

建议保留：

- Discord 用户身份
- 历史最高塔层
- 历史最高名次
- 轮回次数

### 播报原则

- 只有获得**传说命格**时才全服播报
- 命格名称本身应体现品质感
- 传说命格应一眼看出更有牌面


---

## 13. 论道榜系统

### 定位

玩家主动打榜，不做自动匹配。

### 结构

- **论道榜**：唯一主动 PvP 榜
- **战力值**：只做展示参考，不单独做可挑战榜
- **战力榜 / 分境界战力榜 / 通天塔榜 / 本命法宝榜 / 境界榜**：作为自动派生榜单，用于展示与虚荣心反馈

### 规则

- 玩家只能挑战自己前方 **5 名** 以内的目标
- 每日挑战次数：**5 次**（按自然日刷新）
- 挑战胜利：直接互换名次
- 挑战失败：消耗次数，不掉自己排名
- 防守方不需要配置阵容，直接读取当前角色面板快照

### 战斗模型

一期极简即可：

- 杀伐
- 护体
- 身法

并遵守：

- PvE 与 PvP 共用同一套战斗核心
- 身法高者先手，同身法随机
- 1 回合 = 双方各行动 1 次
- 最多 10 回合
- 达到回合上限仍未分胜负时，主动挑战方失败

禁止实现：

- 技能树
- 手动出招
- 职业体系
- 复杂元素克制

---

## 14. 榜单、称号与虚荣心

一期至少做这些“能晒”的内容：

- 当前境界
- 当前命格
- 当前称号
- 论道榜当前名次
- 历史最高名次
- 最高塔层
- 最近一次高光事件

### 榜单建议

一期建议至少保留：

1. 论道榜
2. 综合战力榜
3. 各大境界战力榜
4. 通天塔榜
5. 本命法宝榜
6. 境界榜

### 称号原则

称号主要提供：

- 面子
- 展示
- 播报价值

不要给过高的数值加成。

其中：

- **论道榜称号**应更中二、更凶、更有压迫感
- **分境界战力榜称号**可保留如“炼气第一人”这类更直观的表达


---

## 15. 播报系统

### 目标

靠文案、emoji、节奏感制造“修仙味”，而不是靠图片或动画。

### 一期做法

只做：

- 结构化事件
- 固定模板或模板池播报

适合播报的事件：

- 传说命格出现
- 大境界突破
- 进入论道榜前列
- 创造历史新高塔层
- 轮回重修结果

### AI 的位置

允许后续接入 AI，仅用于：

- 润色高光播报文案

不允许 AI 参与：

- 数值计算
- 战斗判定
- 掉落判定
- 奖励判定
- 排名逻辑

## 16. 推荐技术栈

一期推荐使用：

- **Python 3.12**
- **discord.py**
- **SQLite**
- **SQLAlchemy 2.x**
- **Alembic**
- **APScheduler**（仅处理少量全局任务）
- **Docker Compose**

更完整的工程约束与未上线阶段规则，见：

- `design/06-技术栈与工程约束.md`
- `design/07-开发骨架与落地顺序.md`

### 不建议一期使用

- PostgreSQL（除非已明确需要多人并发扩张）
- Redis
- Celery / MQ
- 微服务拆分
- 前后端分离
- 自建 Web 后台
- 在未正式上线阶段提前做兼容层、迁移适配层、过渡结构

## 17. 推荐项目结构

建议保持轻量：

```text
bot/
  main.py
  config.py
  db.py
  models/
  services/
  commands/
  views/
  scheduler/
  utils/
```

### 建议模块职责

- `models/`：数据库模型
- `services/`：玩法逻辑（登塔、挂机、突破、命格、论道）
- `commands/`：Slash commands 入口
- `views/`：Discord 按钮和面板 View
- `scheduler/`：少量全局定时任务
- `utils/`：进度条、文案、格式化、稀有度颜色等

## 20. 开发行为约束（给 AI）

在实际编码时，必须遵守：

1. **每次只做一个小目标**，不要一口气重构全项目
2. 新增功能前，先确认是否真在一期范围内
3. 能写死在代码里的配置先写死，不要过早配置化
4. 优先做可运行、可验证的最小版本
5. 任何“为了以后扩展”的大型抽象都应默认拒绝
6. 任何涉及图片生成、复杂前端、复杂调度、复杂经济系统的提议都应默认拒绝
7. 如果某个实现明显增加复杂度，但没有直接提升一期可玩性，应放弃

### 允许的工程折中

- 少量重复代码可以接受
- 少量硬编码可以接受
- 某些数值先写在 Python 常量里可以接受
- 某些文案先用模板字符串可以接受

### 不允许的借口

- “为了后续多平台支持”
- “为了以后加更多玩法”
- “为了做成通用框架”
- “为了让数值可配置”
- “为了避免未来重构”

如果当前需求并不需要，请不要实现。

---

## 21. 一期验收标准

当满足以下条件时，可认为一期可玩：

1. 玩家能通过 `/修仙` 创建角色并看到公开面板
2. 玩家有随机命格，且命格只提供一条小加成
3. 挂机收益可正常随时间补算
4. 气机可正常恢复并影响登塔次数
5. 玩家可以登塔、卡关、冲新高
6. 玩家可以突破，且突破受修为与守关层双条件限制
7. 玩家有一件本命法宝，且能提供有限成长反馈
8. 玩家可以挑战论道榜并互换名次
9. 榜单、称号、播报能提供基本虚荣反馈
10. 系统整体运行稳定，不需要复杂后台即可维护

## 22. 最后的提醒
