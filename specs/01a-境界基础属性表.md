# 01a-境界基础属性表

> 状态：Draft v2（首版完整表）
> 作用：承载境界基础数值表，定义 36 个固定成长阶段的修为上限与基础三维。
> 关联文档：
> - `design/specs/01-角色与基础成长规格.md`

---

## 1. 文档作用

本表是整个一期数值系统的基础地基，负责定义：

- 每个大境界 / 小境界的显示名
- 每个阶段的修为上限
- 每个阶段的基础杀伐
- 每个阶段的基础护体
- 每个阶段的基础身法

后续系统都必须引用本表，而不是各自发明一套平行数值。

---

## 2. 字段结构

每一行对应一个固定的成长阶段（共 36 行）。

建议字段如下：

| 字段名 | 含义 |
|---|---|
| `realm_key` | 大境界内部键，例如 `lianqi` |
| `realm_index` | 大境界顺序，1~9 |
| `realm_name` | 大境界显示名，例如 `炼气` |
| `stage_key` | 小境界内部键，例如 `early` |
| `stage_index` | 小境界顺序，1~4 |
| `stage_name` | 小境界显示名，例如 `前期` |
| `display_name` | 完整显示名，例如 `炼气·前期` |
| `global_stage_index` | 全局阶段序号，1~36 |
| `cultivation_max` | 当前阶段修为上限 |
| `base_atk` | 当前阶段基础杀伐 |
| `base_def` | 当前阶段基础护体 |
| `base_agi` | 当前阶段基础身法 |
| `reinforce_cap` | 当前阶段所属大境界的法宝强化上限 |

说明：

- `reinforce_cap` 虽然可由 `realm_index * 10` 推导，但放在表中可读性更好。
- 逻辑实现时可选择运行时推导，不强制必须存入数据库。

## 5. 全局阶段映射规则

```text
global_stage_index = (realm_index - 1) * 4 + stage_index
```

示例：

- 炼气·前期 = 1
- 炼气·圆满 = 4
- 筑基·前期 = 5
- 结丹·中期 = 10
- 渡劫·圆满 = 36

该序号适合用于：

- 成长公式
- 曲线插值
- 榜单辅助比较
- 塔层门槛映射

## 6. 数值设计原则

### 6.1 基础三维原则

- `base_atk`、`base_def`、`base_agi` 全部随 `global_stage_index` 递增
- 不允许出现同阶段回退
- 每个大境界切换时应有明显跃升感
- 同一大境界内的前 / 中 / 后 / 圆满也应有稳定增长

### 6.2 修为上限原则

- `cultivation_max` 随阶段递增
- 同一大境界内应逐步增长
- 跨大境界时可有更明显跳升
- 建议先做可手调的表，不要追求过于复杂的公式美感

### 6.3 三维分布原则

一期建议保持以下倾向：

- `base_atk` 稍高于 `base_def`
- `base_def` 稍高于 `base_agi`
- `base_agi` 不能低到毫无存在感

因为：

- 杀伐是稳定输出轴
- 护体是稳定生存轴
- 身法虽然带波动，但仍必须有足够基础数值承载其价值

### 6.4 法宝强化联动原则

后续法宝强化成长值应当根据本表反推，满足：

- 在正常培养下，本命法宝总贡献长期约占角色总战力的 15%
- 不能让法宝成长压过境界基础属性
- 同一大境界内，法宝成长能带来明确分化，但不应超出境界压制

## 7. 数值风格拍板（v2）

### 7.1 修为曲线风格

采用：

> **走修仙膨胀路线，但控制凡界终盘规模；前期平缓，中后期拉开，并允许关键大境界切换出现量级断层。**

解释：

- 炼气、筑基、结丹阶段的修为门槛增长相对温和
- 在 **结丹 → 元婴** 允许出现一次明显量级跃升
- 在 **化神 → 炼虚** 允许再出现一次明显量级跃升
- 但凡界一期的终点（渡劫）不应过早冲到过高数量级，需为后续可能扩展的仙界篇（伪仙 / 真仙 / 金仙等）保留充足空间
- 到渡劫时，修为量级建议控制在“亿”这一常用中文数量级区间内，不提前逼近“万亿”等更高量级

### 7.2 显示层数字格式

由于修为与后续部分战斗派生数值将走大数字路线，显示层必须统一提供中文单位格式化。

建议统一采用：

- `< 1万`：原样显示
- `>= 1万 且 < 1亿`：显示为 `x.xx万`
- `>= 1亿 且 < 1万亿`：显示为 `x.xx亿`
- `>= 1万亿`：显示为 `x.xx万亿`

建议：

- 默认保留 1~2 位小数
- 去掉无意义尾零

### 7.3 基础三维曲线风格

采用：

> **基础三维也跟随境界膨胀，但膨胀幅度低于修为；跨大境界有明显跃升，同一大境界内部平滑增长。**

### 7.4 基础三维比例风格

一期建议默认保持：

> **杀伐 > 护体 > 身法**

但差距不要过大。

## 9. 本版数值观察与调参重点

本版完整表的特点：

- 炼气 → 结丹：增长平滑，便于玩家建立直觉
- 结丹 → 元婴：出现第一次明显量级跃升
- 化神 → 炼虚：出现第二次明显量级跃升
- 合体 / 大乘 / 渡劫：进入终盘，但增长速度刻意收稳，避免过早炸穿显示层

后续若需要继续微调，优先关注：

1. 元婴前后的跃升感是否足够
2. 化神 → 炼虚的断层是否需要再放大
3. 合体 / 大乘 / 渡劫后段是否需要更陡或更缓

---

## 10. 与法宝强化的联动草案

后续反推法宝强化成长区间时，建议遵守：

- 同一大境界内，法宝强化能提供明显分化
- 但法宝总贡献长期应收敛在约 15%
- 强化上限必须继续被大境界卡住，不能脱缰

建议调参思路：

1. 以本表的 `base_atk / base_def / base_agi` 为主轴
2. 先粗估同境界满强化时的法宝总增量
3. 若法宝贡献明显逼近或超过境界基础属性，则应回调成长值

## 11. 非目标 / 禁止混入内容

本表只承载境界基础属性，不要往里混入：

- 挂机收益
- 塔层奖励
- 命格加成
- 论道榜挑战次数
- 播报模板
- 战斗日志文案
- 强化成功率分段表

一句话：

> **01a 是“地基表”，不是全系统数值杂烩。**
