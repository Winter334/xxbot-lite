# 06-技术栈与工程约束

> 作用：给 IDE / AI 编码助手一个**不会走歪的工程边界**。本文档不是讨论“什么都不能做”，而是直接拍死一期开发方向，避免 AI 自作聪明上重型方案。

---

## 1. 一句话工程定义

这是一个：

> **单进程、单仓库、SQLite 持久化、基于 Discord 交互的文字修仙 Bot。**

一期目标是：

- 先做出可玩闭环
- 先把主流程跑通
- 先让规则真正落地

不是：

- 架构展示项目
- 通用 bot 平台
- 可无限扩展的游戏后端框架
- “未来也许会用到”的抽象练习

---

## 2. 技术栈拍板

一期默认技术栈直接固定为：

- **Python 3.12**
- **discord.py**
- **SQLite**
- **SQLAlchemy 2.x**
- **Alembic**
- **pytest**
- **Docker Compose**

若确有少量定时需求，可使用：

- **APScheduler**

但应优先使用：

- 懒结算
- 玩家交互时补算

而不是依赖高频调度。

---

## 3. 工程方向约束

### 3.1 部署形态

一期默认就是：

> **一个 Bot 进程 + 一个 SQLite 文件。**

不要擅自改成：

- 多服务
- 多进程编排
- Redis + Worker
- MQ + 异步任务链
- 外部缓存层

### 3.2 数据流原则

一期优先：

- 直接函数调用
- 直接 service 调 service
- 直接数据库读写

不要为了“以后扩展”先做：

- 事件总线
- 领域事件系统
- 插件机制
- 通用规则引擎
- DSL 数值系统
- Buff / Effect 引擎

### 3.3 未上线阶段的特殊硬约束

当前项目仍处于：

> **未正式上线、无历史包袱、无旧数据用户阶段。**

因此在这个阶段：

- **不主动做兼容层**
- **不主动做迁移适配**
- **不主动做旧版本数据清洗逻辑**
- **不主动做回滚兼容写法**
- **不主动做双轨字段 / 双轨接口 / 过渡结构**

原则：

> **没有真实历史负担，就不要发明历史负担。**

如果当前需求变了，直接改现有实现；不要为了一个还不存在的“旧版本用户群”提前背包袱。

### 3.4 接口与抽象原则

一期应优先写：

- 明确、短小、可追踪的 service
- 明确、短小、可追踪的 command handler
- 足够用的数据表与配置表

不优先写：

- 通用父类体系
- 复杂抽象层
- 可插拔适配器生态
- Provider 架构
- 将来可能接网页前端的接口层预埋

---

## 4. AI 编码助手执行原则

如果你是 IDE 内的 AI 编码助手，请强制遵守：

1. **优先做能直接跑通的最小实现**
2. **优先把规则写死在一期范围内**
3. **优先让代码短、直、可查**
4. **优先复用现有文档已拍板的规则，不自行脑补扩展系统**
5. 若某个设计的主要价值是“以后可能方便”，但对一期可玩性没有直接帮助，则不实现
6. 若某个设计会增加文件数量、抽象层数、概念数量，但不能立刻提升一期可用性，则不实现
7. 当前没有真实历史包袱时，不写兼容逻辑、不写迁移兜底、不写过渡层

---

## 5. 判断标准

当你犹豫某个实现要不要做时，只问这一个问题：

> **它是否直接帮助“一期玩家现在就能玩到更完整的主循环”？**

- 如果答案是 **是**，可以做
- 如果答案是 **不是，只是以后可能有用**，不要做

---

## 6. 最终约束

一句话总结：

> **一期开发方向是：轻、直、够用、能跑。不要替未来上线版本提前背兼容债，也不要替未来大版本提前造框架。**
