# 07-开发骨架与落地顺序

> 作用：固定一期开发顺序与骨架，避免 IDE / AI 编码助手一上来乱铺系统、乱补未来功能、乱拆工程。

---

## 1. 一期开发目标

一期要做的不是“功能看起来很多”，而是：

> **尽快做出一个能从创建角色一路玩到挂机、登塔、突破、论道、榜单展示的最小可玩闭环。**

判断标准不是“架构是否优雅”，而是：

- 能不能创建角色
- 能不能打开面板
- 能不能稳定推进主线
- 能不能打榜
- 能不能产生牌面

---

## 2. 建议目录骨架

建议优先保持单仓库、单应用、低层数目录：

```text
project/
  bot/          # Discord 入口、启动、事件绑定
  commands/     # Slash commands 与按钮交互入口
  services/     # 核心玩法逻辑
  models/       # SQLAlchemy 模型
  data/         # 静态配置表（境界表、命格表等）
  ui/           # Embed / 按钮渲染
  tests/        # 基础测试
  migrations/   # Alembic
```

原则：

- 不必为了“分层漂亮”再额外套更多壳
- 先让人能顺着目录一眼看懂主流程

---

## 3. Service 划分建议

一期 service 只建议先有这些：

- `character_service`
- `idle_service`
- `tower_service`
- `breakthrough_service`
- `combat_service`
- `ladder_service`
- `ranking_service`
- `fate_service`
- `artifact_service`

说明：

- `combat_service` 是唯一战斗核心
- `tower_service` 与 `ladder_service` 都调用 `combat_service`
- 不要再额外拆出第二套 PvP / PvE 战斗层

---

## 4. 推荐开发顺序

### 阶段 1：项目骨架

先完成：

- Discord bot 启动
- SQLite + SQLAlchemy 接通
- Alembic 初始化
- 基础目录与配置加载

目标：

- 项目能启动
- 数据库能建表
- 基础命令能响应

### 阶段 2：角色创建与主面板

先完成：

- 创建角色
- 初始命格分配
- 初始法宝分配
- `/修仙` 主面板
- 他人面板查看（可后置，但结构先预留）

目标：

- 玩家第一次进入就能拿到角色并看到完整核心信息

### 阶段 3：挂机修炼与气机

先完成：

- 挂机补算
- 气机恢复
- 面板中的修为 / 气机 / 当前挂机层展示

目标：

- 不操作也能成长
- 面板能体现“我一直在变强”

### 阶段 4：通天塔与突破

先完成：

- 塔怪生成
- 塔战结算
- 新高层记录
- 首通奖励
- 突破判定与突破执行

目标：

- 主线成长闭环跑通

### 阶段 5：论道榜与 PvP

先完成：

- 论道榜排名结构
- 每日前 5 名挑战
- 排名互换
- PVP 战报输出

目标：

- 主动竞争闭环跑通

### 阶段 6：自动榜单与称号

先完成：

- 综合战力榜
- 分境界战力榜
- 通天塔榜
- 本命法宝榜
- 主称号 / 荣誉标签展示

目标：

- 虚荣心系统开始发力

### 阶段 8：本命法宝词条与洗炼

在主线闭环稳定、手动闭关与法宝强化已落地后，再补：

- 法宝词条槽位
- 词条洗炼面板
- 词条 pending / 保存逻辑
- 词条触发框架
- 首批词条池

目标：

- 为强化满后的玩家提供新的成长出口
- 让本命法宝形成构筑差异
- 继续围绕本命法宝扩展，而不是立刻引入装备系统

### 阶段 9：播报与轮回重修

最后补：

- 传说命格播报
- 高光播报
- 轮回重修

目标：

- 包装体验完整
- 长期目标出现

---

## 5. 开发时的硬约束

如果你是 AI 编码助手，请严格遵守：

1. **按阶段顺序做，不要跳着铺系统**
2. 没轮到的阶段，最多只留最小接口，不要先把完整系统写出来
3. 没有被当前阶段直接使用的抽象，不要提前创建
4. 当前未正式上线，没有历史包袱时，不要主动写兼容层、迁移适配层、过渡结构
5. 如果某个功能只是“以后可能会有”，但本阶段没有直接调用点，就先不做

---

## 6. 最小闭环优先级

一期开发顺序中，优先级最高的是：

> **先让玩家能从“创建角色 → 变强 → 冲塔 → 突破 → 打榜”完整走通。**

只要这个闭环还没跑顺，就不要把时间花在：

- 后台管理页
- 通用配置平台
- 复杂埋点
- 未来多服支持
- 未来数据兼容
- 未来扩展接口

---

## 7. 最终约束

一句话总结：

> **先做主线闭环，再补虚荣系统，再补包装；不要一开始就把“以后可能会有的东西”提前做出来。**
