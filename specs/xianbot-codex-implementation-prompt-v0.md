# 修仙 BOT 命格/阵营实现提示词（给本地 Codex）v0

你正在为一个已有的 Discord 修仙 Bot 项目实现新系统。请严格遵守以下要求：

## 你的工作目标
根据仓库中的设计文档，帮助实现：
- 阵营系统（正道 / 魔道、悬赏、劫掠、善恶、气运、逆天改命相关支撑）
- 命格系统重做所需的底层扩展与首批命格池

## 必须先阅读的文档
请先阅读并理解以下文档，再开始动手：
- `design/xianbot-faction-system-v0.md`
- `design/xianbot-fate-mechanism-whitelist-v0.md`
- `design/xianbot-fate-pool-v0.md`

必要时请同时检查代码中的现有实现，尤其是：
- 命格定义与命格服务
- 战斗结算逻辑
- 资源结算逻辑（修为、器魂）
- 面板与 UI 入口

## 最重要的行为规则
### 1. 遇到关键不明确处，必须停下来提问
不要为了继续实现而自行猜测设计意图。

以下情况一旦遇到，必须暂停并向我提问：
- 文档只描述方向，没有明确到实现规则
- 某个效果需要新增机制，但文档没有说清楚机制细节
- 某个数值会明显影响平衡，但文档未给出明确口径
- 某个现有实现与文档冲突
- 你认为存在两种以上都说得通的实现方式
- 你需要为了方便实现而修改设计含义

### 2. 不要边猜边干
禁止以下行为：
- 默认把模糊文案替换成你自认为合理的规则
- 未沟通就擅自删改命格效果
- 为了省事，把需要讨论的新机制简化成完全不同的东西
- 把文档中“待进一步明确”的部分直接编码成拍脑袋方案

### 3. 可以主动提出实现建议，但不能越权拍板
如果你发现某个设计不好实现，可以：
- 明确指出问题
- 给出 1~3 个可选实现方案
- 说明各自优缺点
- 等我确认后再继续

## 实现优先级原则
1. **优先复用现有系统**
2. **只在必要时新增通用机制**
3. **避免为单个命格写特例逻辑**
4. **尽量通过配置化/表驱动扩展命格**
5. **任何复杂机制都要先确认再写**

## 关于命格系统的特殊要求
- 命格不是普通小词条，效果要有存在感
- 命格尽量不做一次性、临时性效果
- 命格优先是长期生效、结构型、倾向型效果
- 如果某条命格需要“浮动区间 / 结果权重 / 保底”之类的新结构，而现有系统没有，请先停下来讨论，不要自己定义
- 如果某条命格实现会让项目新增很多状态管理、复杂时序或专属机制，请先停下来讨论

## 关于输出方式的要求
当你开始实现时，请按下面格式汇报：
1. 你理解的目标
2. 你准备修改的文件
3. 你发现的疑点 / 待确认点
4. 在我确认前，不要直接实现存在关键歧义的部分

如果你已经完成某一小步，请汇报：
- 改了什么
- 为什么这样改
- 哪些地方仍然需要我拍板

## 一条硬规则
**宁可停下来问，也不要靠猜补全设计。**

如果你发现文档没有覆盖到某个关键细节，请明确告诉我：
- 缺的是哪条规则
- 不同实现会造成什么差异
- 推荐我先决定什么
